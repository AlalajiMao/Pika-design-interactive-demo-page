What exactly is an "App"?

The whole design hangs on one question: what exactly is an "App"? Build one and see. Anime Character Creation — four beats for the user, three steps in the spec; the pick is a binding, not a step. Press Next and watch UI become data: an App is not code — it's the document taking shape on the right.

  1. Type a text prompt describing the character.
  2. Get four candidate faces; pick one.
  3. Fine-tune facial parts (eyes, hair, jawline) in an editor.
  4. Describe the motion; get a short video of the character.

plug = the typed value a step produces · socket = the input type it accepts — a block only fits if the types match, checked while assembling (Appendix B)

The same document also covers one-click Apps. A step waits for a human only because its input_mapping binds $user.* — collect every input up front, bind each later step to prior outputs, and the spec runs end to end unattended. Pacing is data, not architecture (design doc §1.2 ↗).
Storage: the spec is a small, immutable JSON document — it simply lives in Postgres as JSONB, in the same database as everything else. Why one database: design doc §5.5, Appendix D.

When it breaks

Six things break in six different places: the provider reports an error, our own side double-fires or crashes, the message in between gets lost, the provider goes down for real — or the user simply changes their mind. Each one is a transition in the same state machine, absorbed by a specific field. One per Next.

The bar: a failure must never cost a duplicate render or a stuck Job. Six domains, one state machine, a handful of load-bearing fields — and every transition is a conditional update, so races have exactly one winner.
the provider errors
1 · The provider says "failed"→the gateway sorts it into three bins, because the right response differs: transient — temporary (a 5xx, a timeout); a retry will likely succeed, so back off and retry automatically, invisible to the user · permanent — bad input or a policy block; retrying can never help, so fail fast with an actionable error · quota — not an error, just "wait": reschedule without burning an attempt.
error_class
our side double-fires
2 · We send the same work twice→not a provider failure — a double-click, a redelivered queue message, or a re-run after a crash can fire the same call twice, and the danger is double billing. The idempotency key collapses identical calls into one billed call. The one exception is deliberate: the user's explicit Regenerate is a new purchase.
idempotency_key
our process crashes
3 · Our own worker dies mid-step→nothing to do with the provider: a worker is killed or a deploy restarts it halfway. State lives in rows and work in queue messages, so the unacknowledged message simply redelivers to another worker — carrying the same idempotency key from #2, which is exactly why the crash costs nothing. Recovery is a property of the shape, not a feature we wrote.
status (monotonic)
the message gets lost
4 · The work finished — but the news never arrived→the provider did the render, but its completion webhook got lost (a network blip, our deploy, their bug) — unguarded, the step would hang "running" forever. So every dispatch sets its own alarm; when it fires we ask the provider directly: still working → wait more · actually finished → collect the result · unknown → timed_out, retried like a transient. Push when it works, verify when it doesn't.
provider_task_id deadline_at
the provider is down, not blipping
5 · A provider dies for an hour→the difference from #1 is timescale: retries absorb minutes of trouble, not an hour — an hour of piled-up retries becomes a thundering herd at recovery. So the circuit breaker opens per primitive and new steps fail fast with a visible "provider degraded", instead of queueing silently; recovery ramps back up with jitter.
gateway circuit + queue
not a failure — the user brakes
6 · The user cancels an expensive render→a cancel must actually stop the meter: pending → dequeue, nothing billed · running → best-effort provider-side cancel; a result arriving late is rejected by the conditional transition. Paid intermediates are kept for reuse.
cancel_requested

Resiliency & Scalability

Both answers have already appeared in the preceding scenes and in the doc — this page pulls them together, against the brief's own wording.

Resiliency

From the brief
"Resiliency. Video generation is expensive and failure-prone. How do you handle long-running jobs, retries, and task-level monitoring?"
Long-running jobs

A run is rows + messages — no process ever holds a job. Every async dispatch schedules its own deadline watchdog, and a failed Job resumes from the last succeeded step's checkpoint, never from the top. (Scene 7)

Retries

The gateway classifies every failure — transient / permanent / quota — and retries with backoff under per-primitive policy; idempotency keys make every retry bill-safe. (Scene 8)

Task-level monitoring

Per-primitive failure & queue-depth metrics, stuck-job SLAs, 10× cost-anomaly alerts — and an ops view where any job is a StepRun timeline with force-retry · force-fail · skip. (Scene 9)

Full detail: design doc §6.1–6.7 ↗

Scalability

From the brief
"Scalability. How could a team like Growth marketing team ship a new App without engineering effort?"
Compose, don't code

Growth assembles Apps on a canvas from registry cards that carry cost and latency; drawing a wire writes a line of spec, and a type-incompatible wire cannot be drawn. (Scene 5)

Nothing publishes on faith

A sandbox dry-run on real calls gates the publish button; publishing appends an immutable version, rolled out gradually — running Jobs stay untouched.

New capability, no UI work

A newly registered primitive appears in the palette automatically; a new kind of step UI is one shared component, added once by engineering, reused by every future App.

The calls we made

None of this was free. Five decisions carry the design, each buying simplicity at a named cost — one per Next.

§5.1 · Sync vs. async executionDoes a user request wait for the model?→ async-first, SSE pushEvery action is a short write that returns immediately. Under ~2s returns inline — per primitive, never per App. Push is an accelerator, not a correctness dependency: if SSE dies, the page degrades to refresh, not to wrong.
§5.2 · State managementWhere does a Job's position live?→ rows + a delayed-job queue (pg-boss) — not a workflow engineWe looked hard at Temporal and said no: the rows must exist anyway (status, idempotency, BI), so a whole-Job engine means keeping two sources of truth consistent — and user-paced Jobs living days across deploys means workflow-versioning discipline forever. What's genuinely hard — timers, backoff, redelivery — a delayed-message queue does off the shelf; riding in the state database buys transactional enqueue for free. Temporal keeps a named entry point: step-level workflows, when fully automated fan-out/join Apps become real. The boundary is the decision; the substrate below it is swappable.
§5.3 · App representationApps: data or code?→ declarative spec, engineering-owned `custom` step as the escape hatchThe expressiveness ceiling is a feature at v1: what Growth can draw, the platform can validate, price, and run safely.
§5.4 · Gateway in front of the model APIsWho talks to the model APIs?→ one gateway, nobody elseEvery call costs real money — the chokepoint is the point: one rate budget per primitive, one dedupe, one cost meter. Stateless replicas, so the failure-domain cost is contained.
§5.5 · Storage layoutOne database, or one per shape?→ one Postgres, schemas per domainSpec volume is single-digit gigabytes and infinitely cacheable; the load that actually grows needs ACID and ClickHouse — a document store solves neither. The rule used throughout: bake in what's expensive to retrofit (idempotency keys, tenant tags); defer what's cheap to migrate (the engine behind an interface).

Prompts to steer a coding agent

One prompt per workstream, issued in build order. These are long tasks, so every prompt is built for planning and multi-session continuity, not one-shot generation.

The strategy

Scope: deliberately just the design doc plus each workstream's initial prompt. Every thread accumulates its own follow-ups — adjustments, tests, debugging — across many commits; that steering belongs to the owning engineer's taste and judgment. The preamble's progress file and one-task-per-commit discipline are what keep those follow-ups anchored to the doc.
The shared preamble — prepended to every prompt, verbatim 
"The design doc is your specification — read the
sections named below before writing code. First, split
this workstream into concrete tasks (a, b, c, …) and
write that plan to progress/<workstream>.md. After each
completed task, update that file: what is done, decisions
made, any deviation from the doc. Work one task per
commit — small, reviewable, tests included; never batch
the workstream into one commit. If the design and reality
conflict, stop and record the conflict in the progress
file instead of silently diverging. A new session starts
by reading the design doc and the progress file, then
continues from the first unchecked task."

§3.1Authoring & publish — Growth creates an App
  1. Engineering stocks both shelves — primitives registered with I/O schemas, cost/latency, quota & retry policy; each kind of step UI added once to the shared component library.
  2. Growth drafts — registry cards load onto the canvas; every drawn connection is saved as an input_mapping line, compatibility checked in the browser.
  3. Growth validates — static checks re-run, then a dry-run on saved fixture inputs through the production gateway on a sandbox quota; a current PASS unlocks publish.
  4. Growth publishes — v(N) appended, immutable, staged rollout; in-flight Jobs stay pinned to their version.
§3.2A user runs an App — the happy path  (animated in Story · Scene 7)
  1. Create — POST /jobs writes the jobs row pinned to the spec version; 201 in milliseconds, nothing generated or billed.
  2. Submit a step — the synchronous half — one transaction inserts the StepRun and enqueues execute_step; 202; the HTTP request ends here.
  3. The queue takes over — a worker calls through the gateway, records the provider's task id, schedules its own deadline check, exits.
  4. The result comes back — the callback lands; one transaction writes assets + lineage, flips the row to succeeded, NOTIFYs; SSE pushes step.completed.
  5. Repeat; finish — the reducer finds no next step → job succeeded. Sync primitives under ~2s return inline.
§3.3Waiting, failure & retry — the StepRun state machine  (animated in Story · Scene 8)
  1. Waiting is a first-class state — running + a deadline from the primitive's p95; reopening the page reads status, SSE resumes.
  2. Transient failure — gateway classifies; retrying with backoff, capped by registry policy; invisible unless attempts run out.
  3. Lost callback — the step's own check_step message verifies at the source: reschedule, recover, or timed_out → retried.
  4. Permanent failure — fail fast with an actionable error; an identical resubmit hits the idempotency key instead of billing again.
  5. Job-level resume — always from the last succeeded StepRun's checkpoint, never from the top.

System · Trade-offs

Five decisions, argued doc §5 ↗

Execution model→ async-first, sync fast path

Decision

Default async: create the StepRun, enqueue in the same transaction, return immediately, push completion over SSE (monotonic seq, fetch-then-subscribe). Primitives under ~2s return inline, per the registry's execution_mode — never per App.

Alternative

Hold HTTP connections open (pinned for minutes, dies on mobile and deploys), or client polling (hammering vs. lag).

Why

Media primitives run seconds to minutes and fail routinely — something has to hold the gap between "user clicks" and "result exists," and that something is rows, not a connection. SSE is an accelerator, not a correctness dependency: if push fails, the page degrades to refresh, not to wrong.

State management→ rows + pg-boss, not a workflow engine

Decision

A Job is a database-backed state machine: Postgres rows are the single source of truth; timers, retries, and redelivery are delayed messages in pg-boss, which rides in the same Postgres — so a state change and the message it triggers commit in one transaction.

Alternative

A durable workflow engine (Temporal) holding whole Jobs as literal sequential code.

Why

The rows must exist anyway (status, idempotency uniqueness, BI), so a whole-Job engine means a second source of truth kept consistent forever — and user-paced Jobs living days across dozens of deploys means workflow-versioning discipline forever. Temporal's named entry point: step-level workflows when fully automated fan-out/join Apps become real; those never wait on humans and rarely outlive a deploy. Job-level sequencing stays in rows permanently.

App representation→ declarative data, not code

Decision

Apps are versioned JSON specs interpreted by a generic runtime; an engineering-owned custom step type is the escape hatch, shipped once, reused by every App.

Alternative

Code-defined workflows on an SDK — maximum expressiveness, but every App becomes an engineering project.

Why

The expressiveness ceiling is a feature at v1: what Growth can draw, the platform can validate, price, and run safely — and what later lets the agent write plans at runtime. Code Apps kill the Growth-ships-alone requirement and are a different risk class for an authoring agent.

Path to the model APIs→ one central gateway

Decision

Nobody calls providers directly. The gateway enforces what the registry declares: auth, per-primitive rate limits, circuit breaking, classified retries, idempotency dedupe — and paces async calls through a bounded per-provider dispatch queue.

Alternative

A shared client library, each service calling providers itself — no extra hop, no chokepoint.

Why

In a system where every call costs real money, the chokepoint is the point: one rate budget, one dedupe, one cost meter. The failure-domain cost is contained because the gateway is stateless and scales as identical replicas.

Storage layout→ one Postgres, schemas per domain

Decision

One managed cluster, split into registry / specs / runtime / agent schemas plus pg-boss's own; spec reads served from the replica (immutable → lag harmless). Jobs pin specs by value, not FK — a pre-cut split line.

Alternative

A document store for the document-shaped spec store from day one, or a cluster per store.

Why

Spec volume grows with authoring rate — single-digit GB, infinitely cacheable. The load that actually grows (StepRuns, events) needs ACID and ClickHouse; a second database is paid for daily on a 1–2-engineer team. Rule: bake in what's expensive to retrofit; defer what's cheap to migrate.

System · Resiliency

Six failure domains, absorbed doc §6 ↗

The bar: a failure must never cost a duplicate render or a stuck Job.

A call fails → classified

The gateway classifies every failure: transient (backoff + jitter as a delayed retry message, attempts capped by registry policy), permanent (fail fast, actionable error), quota (reschedule, don't burn an attempt). Fan-out steps get partial-success via the step's optional min_outputs: deliver at the declared minimum, retry only the shortfall — never re-bill successes.

Same work twice → idempotency

Every Job and StepRun carries project_id + step_id + params_hash; the gateway dedupes, so redeliveries, double-clicks, and crash-restart re-calls collapse into one billed call. The user's explicit Regenerate appends a nonce — a new render on purpose is a new purchase.

Process dies → redelivery

Handlers are stateless; every transition is a conditional update (WHERE status='running'); unacked messages redeliver with the same key. Jobs resume from the last succeeded StepRun's checkpoint, never from the top; intermediates stay in cold storage 30 days.

Lost callback → deadline check

Every async dispatch schedules its own check_step message at p95 × margin. When it fires: still working → reschedule; finished → recover through the same completion transaction; unknown → timed_out, retried as transient. The callback/watchdog race has one winner — the conditional update.

Provider outage → circuit + queue

Retries absorb minutes; an hour needs a different posture. The breaker opens per primitive and the bounded dispatch queue stops admitting — new steps fail fast with a visible "provider degraded." Recovery drains at the provider's declared pace with jittered ramp-up.

User cancels → stop the meter

pending → dequeue, nothing billed. running → cancel_requested + best-effort provider-side cancel; a result arriving after cancellation is rejected by the conditional transition. Succeeded intermediates are kept — paid for, reusable.

Task-level monitoring — and the handles to act on it

Metrics per primitive (failure rate, queue depth, latency vs. estimate), per Job (stuck-past-SLA), per App (cost anomaly — a 10× burn is a retry loop or a bad publish). The ops view renders any Job as its StepRun timeline — status, attempts, error class, provider task id — with three manual actions: force-retry · force-fail · skip step. A 3 a.m. page ends in a click, not a psql session.

Data · Storage

The shape of the data picks the engine doc §4.2 · §5.5 ↗

Transactional state machines Postgres

Jobs / StepRuns: multi-row atomic transitions, DB-enforced idempotency uniqueness, monotonic statuses — the correctness load-bearing wall. pg-boss rides in the same cluster, so a state change and the message that drives it commit atomically.

Everything else, by shape

Immutable documents (specs, registry) → Postgres JSONB, replica-served · bytes → S3 + CDN · append-only aggregates → ClickHouse · ephemeral pacing state → in-memory/Redis, deliberately lossy.

v1: one managed Postgres cluster, schemas per domain (registry · specs · runtime · agent + pg-boss). Pre-cut split line: Jobs pin specs by value — registry/specs can move clusters with a copy job and a connection-string change. Full argument: design doc §4.2, §5.5, Appendix D.

The loop, each turn

1

Assemble context

The five-layer recipe — compacted history, asset manifest, budget state, tool catalog. Assets travel by reference: one vision description at creation, reused forever.

2

LLM decides

Call a tool, ask the user, or present results. Top-tier model plans; a small model handles mechanical turns. Human checkpoints are first-class exits, not failures.

3

Execute via the Job Runner

Tool calls pass the budget gate and append StepRuns to the session's open Job — never direct API calls. Idempotency, retries, budgets, resume: zero extra plumbing.

4

Observe & repeat

Asset refs + cached descriptions return to the loop — until intent is met, a budget gate asks first, a loop guard trips, or the user cancels.

Cost · media spend dominant §15 ↗

  • Budget envelopes per session and per turn, estimated pre-call from registry cost_estimate — checked and deducted in the same transaction that creates the StepRun.
  • Confirmation gates above a threshold (any video render) — estimate shown, agent waits.
  • Draft-first: low-res proxies and single keyframes; full render only after approval.
  • Part 1 idempotency already prevents double-billing on retries.

Cost · LLM tokens compounding

  • Model tiering — the top model plans; a small model executes mechanical turns.
  • Prompt caching on the stable system prompt + tool catalog.
  • Compaction — sessions stay O(recent turns); the summary is derived, always rebuildable.
  • Loop guards — a thrashing agent stops and asks instead of burning budget.

Testing & evaluation release gate §16 ↗

  • Harness tests (CI): the machinery with the LLM removed — recorded responses + mocked primitives, deterministic, every commit.
  • Tool-selection evals (prompt gate): context fixtures replayed against the live planner only — calls validated, never executed; tokens-only cost gates every prompt/tool-description/model change.
  • Offline suite (release gate): 50–100 briefs × N runs → score distributions; programmatic checks + a pinned LLM judge calibrated against human scores; median regression blocks deploy.
  • Pairwise comparison for prompt/model changes — shared noise cancels.
  • Online: canary by session %; draft acceptance, regens per asset, cost per session — plus explicit 👍/👎 that recalibrates the judge and seeds new briefs.
